1. 传输通道

通道 用途 入口/路径 数据格式
BLE GATT 蓝牙配网、蓝牙唤醒 Wi-Fi Service/Characteristic 见下文 ASCII JSON 写入特征值，JSON/嵌套 JSON 通知或读取
HTTP 投图、保存图片管理、轮播、保活 http://<device-ip>/dataUP 上传用 multipart/form-data，控制命令用 JSON
HTTP GET 连通性校验 http://<device-ip>/ping 无请求体，返回 JSON
  • 设备返回 result: 0 或 result: "0" 表示成功。
  • 部分 BLE 响应会包一层 { "payload": "{...}" }，前端会优先解析 payload。
  • HTTP 非 2xx、响应不可解析、超时都按失败处理。

2. HTTP 图片与控制协议

2.1 接口总览

func 用途 请求格式 关键字段
cast 普通投图 POST /dataUP + multipart fileName bin bin_size image image_size save show
upload 通用上传 POST /dataUP + multipart 同 cast
cast2pic 双图投图/上传 POST /dataUP + multipart 与 upload 多张上传一致，重复两组 fileName bin bin_size image image_size，并带 screen
update 替换旧图片，当前前端预留 POST /dataUP + multipart oldfileNames newfileNames bin image
get_saved_images 获取已保存图片列表 POST /dataUP + JSON 无
get_snapshot 获取已保存图片列表与轮播状态 POST /dataUP + JSON images slideshow
start_slideshow 下发轮播列表 POST /dataUP + JSON fileNames interval random
set_slideshow 设置轮播开关与周期 POST /dataUP + JSON sw interval random
set_wifi_work_time 设置 Wi-Fi 工作时长 POST /dataUP + JSON seconds
delete 删除已保存图片 POST /dataUP + JSON fileNames
ping 检查设备网络连通性 GET /ping 无

2.2 cast 普通投图 multipart

POST /dataUP HTTP/1.1
Content-Type: multipart/form-data

func=cast
fileName=26422
bin_size=123456
image_size=23456
save=true
show=true
bin=@26422.bin
image=@26422.jpg

设备端需要校验字段完整性、文件大小和主图/缩略图关联关系。cast 成功后应记录最后一次投图，设备重启或 OTA 后优先显示该图片。

{
  "func": "cast_result",
  "result": 0
}

2.3 upload 通用上传 multipart

upload 与 cast 字段一致，用于保存图片上传。多张图片上传时按 cast2pic 一样重复追加多组同名字段。

POST /dataUP HTTP/1.1
Content-Type: multipart/form-data

func=upload
fileName=26422
bin_size=123456
image_size=23456
save=true
show=false
bin=@26422.bin
image=@26422.jpg

2.4 cast2pic 双图投图 multipart

用于一次请求上传两张完整图片。传输格式与 upload 多张上传完全一致：每张图片都包含一组 fileName、bin_size、image_size、bin、image，两张图片就在同一个 multipart 请求体里重复追加两组同名字段，设备端按字段出现顺序配对。

POST /dataUP HTTP/1.1
Content-Type: multipart/form-data

func=cast2pic
screen=ab
save=true
show=true
fileName=26422
bin_size=123456
image_size=23456
bin=@26422.bin
image=@26422.jpg
fileName=26423
bin_size=123000
image_size=23000
bin=@26423.bin
image=@26423.jpg
  • cast2pic 除 func 外，字段组织方式与 upload 多张上传一致。
  • screen 为字符串更新范围：ab 更新 A+B，a 只更新 A 屏，b 只更新 B 屏；缺省按 ab 处理。
  • fileName、bin_size、image_size、bin、image 都重复出现两次。
  • 设备端按出现顺序把第 1 组主图和缩略图配对为第 1 张，把第 2 组配对为第 2 张。
{
  "func": "cast2pic_result",
  "result": 0
}

2.5 update 替换图片 multipart

当前前端预留。用于用新图片替换旧图片，字段中同时带旧文件名和新文件名。

POST /dataUP HTTP/1.1
Content-Type: multipart/form-data

func=update
oldfileNames=12345
newfileNames=67890
bin_size=123456
image_size=23456
save=true
show=true
bin=@67890.bin
image=@67890.jpg

2.6 get_saved_images 获取已保存图片

{
  "func": "get_saved_images"
}
{
  "func": "get_saved_images_result",
  "result": 0,
  "images": [
    {
      "fileName": "26422",
      "thumbnailUrl": "/thumb/26422.jpg"
    }
  ]
            }

前端会用设备 baseUrl 拼接相对缩略图地址。

2.7 get_snapshot 获取已保存图片与轮播状态

新接口用于一次取回设备本地保存图片和当前轮播配置。images 字段沿用 get_saved_images 的返回数据，slideshow 字段返回设备当前保存的轮播状态。

{
  "func": "get_snapshot"
}
{
  "func": "get_snapshot_result",
  "result": 0,
  "images": [
    {
      "fileName": "26422",
      "thumbnailUrl": "/thumb/26422.jpg"
    },
    {
      "fileName": "26423",
      "thumbnailUrl": "/thumb/26423.jpg"
    }
  ],
  "slideshow": {
    "sw": 1,
    "fileNames": ["26422", "26423"],
    "interval": 60,
    "random": false
  }
}
  • images 与 get_saved_images_result.images 结构一致。
  • slideshow.sw 返回当前轮播开关，1 开启，0 关闭。
  • slideshow.fileNames 返回最近一次 start_slideshow 下发的轮播图片列表，按设备实际轮播顺序返回。
  • slideshow.interval 返回最近一次 start_slideshow 或 set_slideshow 保存的轮播间隔。
  • slideshow.random 返回最近一次 start_slideshow 或 set_slideshow 保存的随机轮播开关。
  • 如果设备未设置过轮播，建议返回 {"sw":0,"fileNames":[],"interval":0,"random":false}。

2.8 start_slideshow 下发轮播列表

{
  "func": "start_slideshow",
  "fileNames": ["26422", "26423"],
  "interval": 60,
  "random": false
}

fileNames 顺序就是轮播顺序；random=true 表示随机轮播；interval 前端会归一为大于等于 1 的整数。

2.9 set_slideshow 设置轮播开关与周期

{
  "func": "set_slideshow",
  "sw": 1,
  "interval": 60,
  "random": false
}

sw=1 开启轮播，sw=0 关闭轮播；random=true 表示随机轮播，random=false 表示按列表顺序轮播。

2.10 delete 删除已保存图片

{
  "func": "delete",
  "fileNames": ["26422", "26423"]
}

2.11 set_wifi_work_time 设置 Wi-Fi 工作时长

{
  "func": "set_wifi_work_time",
  "seconds": 300
}

seconds 前端会归一为大于等于 1 的整数。

2.12 ping 连通性检查

GET /ping HTTP/1.1
{
  "func": "ping_result",
  "result": 0,
  "message": "ok",
  "Ble_MAC": "AABBCCDDEEFF"
}
前端写操作前会优先访问缓存端点的 /ping。如果响应包含 Ble_MAC 或 ble_mac，必须与目标设备 MAC 一致。

3. 蓝牙配网协议

3.1 当前主流程

扫描目标 按设备 MAC 扫描相框广播
Service UUID 7B12FF00-4413-49C1-A307-74997B8B5941
Characteristic UUID 7B12FF14-4413-49C1-A307-74997B8B5941
写入方式 优先 write，否则 writeNoResponse，默认 write
Notify 同一个特征值开启 notify，开启失败时仍继续写入
MTU 尝试设置 256，失败不阻断
超时 总响应 15 秒；第一段成功后等待最终 IP 最多 5 秒

3.2 写入与响应

写入 payload：

{
  "func": "wifi",
  "ssid": "Office-WiFi",
  "key": "password"
}

响应可以直接返回 JSON，也可以放在 payload 字符串中：

{
  "result": 0,
  "message": "find wifi",
  "stage": "Office-WiFi"
}
{
  "result": 0,
  "message": "network ready",
  "stage": "192.168.1.88",
  "version": "3.0.0"
}
  • message=find wifi 且 stage 非空时，stage 保存为已连接 Wi-Fi 名。
  • stage 中能提取 IPv4 地址时，配网最终成功。
  • 最终成功后缓存 baseUrl=http://<ip> 和 commandUrl=http://<ip>/dataUP。

4. 蓝牙唤醒 Wi-Fi

HTTP 投图或图片管理前，如果缓存 HTTP 端点不可用，前端会通过 BLE 唤醒设备 Wi-Fi 并解析新的上传端点。

4.1 连接参数

扫描目标 目标设备 MAC，统一成 AA:BB:CC:DD:EE:FF
扫描超时 10 秒
连接尝试 最多 3 次，每次 4.5 秒，间隔 800 ms
Service UUID 7B12FF00-4413-49C1-A307-74997B8B5941
Characteristic UUID 7B12FF14-4413-49C1-A307-74997B8B5941
读写 写入同一特征值；写入成功后每 800 ms 读取一次同一特征值
响应超时 20 秒

4.2 写入与响应

写入 payload：

{
  "func": "wifi_wakeup"
}

成功响应：

{
  "result": 0,
  "stage": "192.168.1.88",
  "message": "network ready",
  "version": "3.0.0"
}

stage 可返回 192.168.1.88、192.168.1.88:8080、http://192.168.1.88 或 http://192.168.1.88/dataUP，前端都会归一化为 http://<host>/dataUP。

  1. 归一化得到 commandUrl。
  2. 请求 GET /ping。
  3. 如果 /ping 返回 Ble_MAC 或 ble_mac，必须与目标 MAC 一致。
  4. 校验通过后缓存端点；校验失败不写缓存。

5. 设备 BLE 广播识别

扫描设备时前端按统一的 localName 命名格式解析设备信息。

5.1 广播字段

localName 长度大于等于 20，示例：

T!2F782D4AADFAS1F@@A
位置 长度 说明
0 1 厂商 ID，当前支持 T、M
1 1 屏类型字符，通过 deviceNameToTypeMap 映射
2..13 12 MAC，不带冒号
14 1 电量字符，(charCode - 33) * 2 后限制在 0..100
15..16 2 固件版本十六进制，解析为 <major>.<minor>
18 1 功耗/充电信息，bit4 为 1 表示高功耗
19 1 合法标识，A 合法，B 非法

非指定扫描时，前端只展示长度大于等于 20、首字母为 T/M、第二位存在于 deviceNameToTypeMap 的设备。

6. 设备端实现要点

  1. BLE 配网和 BLE 唤醒建议统一实现 Service 7B12FF00... 与 Characteristic 7B12FF14...。
  2. 配网写入 func=wifi 后，设备至少返回一条成功 JSON；若能返回包含 IP 的最终响应，前端会自动缓存 HTTP 端点。
  3. 唤醒写入 func=wifi_wakeup 后，设备必须在 20 秒内返回 result=0 且 stage 为可解析的 IPv4 host。
  4. HTTP 服务端统一接受 /dataUP JSON 控制命令和 multipart 图片上传。
  5. /ping 建议返回 Ble_MAC，用于防止缓存 IP 指向错误设备。
  6. 所有 JSON 响应建议使用 UTF-8，不要包含额外日志文本。
